## 1. 게스트 긁기 플로우 (ScratchCard.jsx + App.jsx)

### 검증 항목

**코드 분석:**
- `ScratchCard.jsx`: 게스트 모드에서 localStorage 플래그 설정 (라인 129)
- `App.jsx`: 게스트 블락 모달로 2회차 로그인 유도 (라인 287-311)
- 80% 전역 임계값 설정 (라인 7-8)

### ✅ 정상 동작
- 1회차 게스트 긁기: 80% 도달 → `showGuestReveal` 모달 표시 (처리됨)
- 2회차: `guest_scratched` 플래그 감지 → 로그인 유도 모달 (처리됨)
- Canvas 은박 애니메이션 후 모달 (800ms 딜레이, 라인 131-134)

### 🐛 발견된 버그

**1. Critical - localStorage 플래그 중복 설정 우려**
- 파일: `ScratchCard.jsx:129` + `App.jsx:332-355`
- 증상: 게스트 긁기 완료 후 `localStorage.setItem('guest_scratched', 'true')` 설정, 그 직후 `handleGuestReveal` 콜백 호출 → 동시에 두 개의 로그인 모달이 실행될 가능성
- 원인: 1. ScratchCard에서 localStorage 설정, 2. App에서 별도 상태(`guestRevealFlyer`) 관리로 인한 이중 처리
- 재현: 빠른 네트워크에서 게스트 긁기 완료 후 2회차 클릭 → 모달이 중첩되거나 이상 동작
- 수정 방안:
  - Option A: localStorage 플래그 제거, App의 guestRevealFlyer 상태만 사용
  - Option B: ScratchCard에서 onGuestReveal 콜백 전에 localStorage 플래그 설정하지 않기

**2. Major - 비로그인 동시 탭 접근 시 localStorage 동기화 문제**
- 파일: `ScratchCard.jsx:129` + `App.jsx:289`
- 증상: 사용자가 여러 탭에서 동시에 게스트 긁기 진행 → localStorage 업데이트는 즉시 반영되지만, App의 상태는 동기화 안 될 수 있음
- 원인: 첫 번째 탭에서 긁기 완료 → localStorage 설정, 두 번째 탭에서는 여전히 handleFlyerClick 실행
- 재현: 같은 게스트 기기에서 여러 탭 열기 → 첫 탭에서 긁기 완료 → 두 번째 탭에서 다른 전단지 클릭
- 권장: `storage` 이벤트 리스너 추가 또는 sessionStorage 사용 고려

**3. Minor - 게스트 회원가입 후 기존 localStorage 미정리**
- 파일: `App.jsx:287-296` (handleFlyerClick)
- 증상: 게스트 긁기 후 로그인하면, localStorage.guest_scratched 플래그가 그대로 유지
- 원인: handleLogin 또는 handleLogout에서 guest_scratched 플래그를 명시적으로 제거하지 않음
- 재현: 게스트로 긁기 → 로그인 → 로그인 유저로 전환되었는데도 플래그가 true
- 수정: `handleLogin` (라인 208-228)에서 localStorage.removeItem('guest_scratched') 추가

  WHERE user_id = ? AND flyer_id = ? AND DATE(verified_at) = DATE('now', 'localtime')

- 원인: Turso는 SQLite 호환 but 시간대 함수는 제한적

- 수정 방안:

  // Option 1: 매개변수로 오늘 날짜 전달
  const today = new Date().toISOString().split('T')[0]
  const existing = await db.prepare(
    `SELECT id FROM visit_verifications
     WHERE user_id = ? AND flyer_id = ? AND DATE(verified_at) = ?`
  ).get(userId, flyer.id, today)

  // Option 2: 백엔드에서 시간 범위 계산
  const startOfDay = new Date()
  startOfDay.setHours(0, 0, 0, 0)
  const endOfDay = new Date()
  endOfDay.setHours(23, 59, 59, 999)

- 원인: 방어 코딩이나 비효율적
- 권장: 백엔드 검증만으로 충분하지만, 프론트에서 UI 표시 위해 필요 (현재 구현 정상)

**3. Minor - 사업자 예산 부족 시 방문인증 미기록**

## 6. 보안 기능 (security.js)

### 검증 항목

### ✅ 정상 동작
- 기기당 최대 2개 계정 제한 (라인 7)
- 긁기 최소 3초 검증 (라인 8)
- 봇 감지 시 세션 무효 처리 (라인 106-115)
- 클라이언트 시간과 서버 시간 비교 (라인 103)

### 🐛 발견된 버그

**1. Critical - 서버-클라이언트 시간 동기화 문제**
- 파일: `security.js:98-103`
- 증상:
  ```javascript
  const serverStartTime = new Date(session.started_at.replace(' ', 'T') + '+09:00').getTime()
  const serverDuration = Number.isNaN(serverStartTime) ? Infinity : Date.now() - serverStartTime
  ```
  SQLite datetime 포맷 ('2026-03-15 12:30:45')을 ISO로 변환하려 하는데:
  - replace(' ', 'T')는 첫 번째 공백만 변환
  - '+09:00' 하드코딩으로 타임존 이슈
  - 만약 started_at이 UTC로 저장되었다면 +09:00 추가 시 부정확
- 원인: SQLite datetime은 기본적으로 UTC 또는 로컬 시간인데, db.js에서 `datetime('now', 'localtime')`으로 저장 (KST 기준)
- 재현: 사용자의 클라이언트 시간이 서버와 크게 다를 때 → 봇 감지 오판
- 수정 방안:
  ```javascript
  // Option 1: ISO 형식으로 변환
  const serverStartTime = new Date(session.started_at).getTime()

  // Option 2: 더 견고한 파싱
  const isoString = session.started_at
    .replace(/(\d{4})-(\d{2})-(\d{2}) (\d{2}):(\d{2}):(\d{2})/, '$1-$2-$3T$4:$5:$6')
  const serverStartTime = new Date(isoString).getTime()
  ```

**2. Major - 클라이언트 시간 조작 취약점**
- 파일: `security.js:103` 및 `ScratchCard.jsx:138-140`
- 증상:
  ```javascript
  const actualDuration = Math.min(serverDuration, clientDuration || Infinity)
  ```
  클라이언트가 긁기 시간을 조작하여 매우 작은 durationMs 전송 불가능하나, 서버 시간으로 재검증 (정상)
  하지만 클라이언트 시간이 몇 초 앞서가면 서버 시간이 마이너스가 될 수 있음
- 원인: clientDuration을 Math.min으로 비교할 때 `|| Infinity` 처리되지만, 어느 쪽이 정확한지 불명확
- 재현: 클라이언트 시간이 서버보다 10초 빠른 경우 → serverDuration이 음수
- 수정:
  ```javascript
  const actualDuration = Math.max(0, Math.min(serverDuration, clientDuration || Infinity))
  ```

**3. Major - scratch_sessions UNIQUE 제약 중복 가능성**
- 파일: `db.js:190` + `security.js:65`
- 증상:
  ```sql
  UNIQUE(user_id, flyer_id)
  ```
  라인 65에서:
  ```javascript
  await db.prepare('DELETE FROM scratch_sessions WHERE user_id = ? AND flyer_id = ?').run(userId, flyerId)
  ```
  DELETE 후 INSERT이므로 race condition 가능성
- 원인: 멀티탭 또는 동시 요청 시 두 번째 startScratchSession이 기존 세션을 덮어쓸 수 있음
- 재현: 같은 전단지를 두 번 빠르게 긁기 시작
- 수정: UNIQUE 제약 제거, token만 UNIQUE 유지하고 DELETE 로직 제거

**4. Minor - Device Fingerprint 약한 해싱**
- 파일: `api/index.js:363-389`
- 증상: `generateDeviceFingerprint()`가 간단한 JS 해싱 사용 (JDK String.hashCode 유사)
- 원인: 실제로는 해싱이 아니라 32비트 정수 체크섬
- 문제: 충돌 확률 높음, 보안 해시 미사용
- 권장: 프로덕션에서는 SHA-256 또는 crypto 라이브러리 사용
  ```javascript
  import crypto from 'crypto'
  const hash = crypto.createHash('sha256').update(raw).digest('hex')
  ```
