# 전단지P 종합 분석 보고서

## 1. 현재 플로우 분석

### 1.1 일반 사용자(User) 플로우

#### A. 온보딩 → 메인 피드 (진입 단계)
- **현황**: App.jsx에서 SPA 라우팅 기반
  - 비로그인 상태: 게스트로 접근 가능 (LoginPage 우회)
  - 소셜 로그인(카카오/구글) 후 역할 선택 모달 표시
  - 디바이스 fingerprint 자동 생성 (다중계정 방지)
- **UX 평가**:
  - 장점: 게스트 모드로 즉시 진입 가능 (이탈 방지)
  - 문제점: 회원가입 시 닉네임+역할 선택이 필수인데, 소셜 로그인 시 닉네임 없음 → 역할 선택 후 닉네임 입력 플로우 불명확

#### B. 전단지 긁기 (핵심 루프)
- **현황**: MainPage → ScratchCard (모달)
  - 비로그인 1회: 게스트 맛보기 (80% 긁으면 당첨 모달, 실제 포인트 미적립)
  - 비로그인 2회: 로그인 강제 모달 (보안/engagement)
  - 로그인 후: scratchToken 기반 서버 검증
- **UX 평가**:
  - 장점: 게스트 맛보기로 초기 engagement 확보
  - **문제점 1**: 게스트 1회 사용 후 "2회차 차단" 모달이 매번 뜸 → 사용자가 "한 번 더 보고 싶은데" 욕구 방해
  - **문제점 2**: 비로그인 상태에서 게스트 블록 모달과 긁기 완료 모달 2개 시스템 → 상태 관리 혼동 가능
  - **문제점 3**: 80% 긩기 threshold는 낮음 (대부분 사용자가 금방 도달) → 기대감 감소

#### C. 퀴즈 참여
- **현황**: DetailPage에서 자동으로 1문제 출제
  - getRandomQuiz(flyerId, userId) 호출 → 이미 응시한 경우 attempted:true
  - 정답 시 +10~50P (문제별 설정)
  - 뮤즈 응시는 회당 1회 제한
- **UX 평가**:
  - 장점: 자동 출제로 friction 없음
  - 문제점: 퀴즈 UI/모달이 빠르게 닫혀서 사용자가 미처 본 후 사라짐 → "뭘 맞혔는지 모르겠는데?"

#### D. QR 방문 인증
- **현황**: QrScanPage에서 BarcodeDetector API 사용
  - 1일 1회 제한 (DATE 기반 체크)
  - 100~500P 적립
  - 사업자 계정은 포인트 불가
- **UX 평가**:
  - 장점: 실제 오프라인 연동으로 신뢰성 높음
  - 문제점: 1일 1회 제한 → 사용자가 같은 가게 여러 번 방문해도 1회만 인정 (불공평 느낌)
  - 문제점: QR 스캔 실패 시 재시도가 번거로움 (하드웨어 의존성)

#### E. 포인트 적립 요약
- **적립 경로**:
  1. 전단지 공유(긁기): sharePoint (자영업자 설정, 평균 10P)
  2. 퀴즈: 10~50P
  3. QR 방문: 100~500P
- **문제점**: 가장 접근성 높은 "전단지 공유"가 가장 적은 포인트 (10P) → 사용자 입장에서 "이 정도면 뭐하러?"

#### F. 기프티콘 교환
- **현황**: GiftShopPage에서 고정 상품 목록 (12개)
  - 최소 3,000P (이디야 아메), 최대 20,000P (도미노피자)
  - 카테고리별 필터링
- **UX 평가**:
  - 장점: 직관적 UI, 전화번호 입력 후 1영업일 내 발송
  - **문제점 1**: 고정 상품만 제공 → "내가 원하는 상품이 없으면?" 교환 불가
  - **문제점 2**: 최소 3,000P 필요 → "전단지 300번 긁어야 1개 교환" (포인트 가치 저평가)
  - **문제점 3**: "교환 신청" 후 카카오톡 도착까지 "최대 1영업일"은 너무 길음 (사용자 기대: 즉시)

#### G. 출금
- **현황**: MyPage 출금 탭
  - 최소 1,000P
  - 7일 경과 필수 (계정 신뢰도 확보)
  - 대기 중인 출금 있으면 중복 신청 불가
- **UX 평가**:
  - 장점: 7일 규칙으로 사기 방지
  - **문제점 1**: "최소 1,000P" → 5개 전단지 긁어야 함 (턴어라운드 시간 길어짐)
  - **문제점 2**: "대기 중 출금 있으면 중복 신청 불가" → 첫 출금 후 "빠른 출금"을 원하는 사용자 좌절
  - **문제점 3**: 출금 상태 조회에 admin_memo가 있는데 → 거부 시 사유 미통지 (사용자 혼동)

### 1.2 자영업자(Business) 플로우

#### A. 전단지 등록
- **현황**: AdminPage의 "flyers" 탭
  - 매장명, 이모지, 색상, 카테고리, 제목, 기간, 공유포인트, QR포인트, 태그, 상품 목록 입력
  - 이미지 업로드 + 크롭 기능
- **UX 평가**:
  - 장점: 종합적 설정 가능
  - **문제점 1**: 폼이 너무 길어서 스크롤 피로도 높음 → 단계별 온보딩이 필요
  - **문제점 2**: "공유포인트" vs "QR포인트"의 개념이 불명확 → 사업자가 어떻게 설정해야 할지 모름
  - **문제점 3**: 이미지 필수가 아니어서 이모지만 사용하는 전단지 많음 → 시각적 품질 저하

#### B. 퀴즈 생성
- **현황**: "quiz" 탭에서 문제별 정답+포인트 설정
  - 각 문제마다 point 설정 (기본 20P)
- **UX 평가**:
  - 문제점: 퀴즈 관리 UI가 별도 탭 → 전단지 등록 후 "아, 퀴즈도 추가해야 하는데" 하게 됨
  - 문제점: 선택지(옵션) 입력 가능 여부 불명확 → 퀴즈 실제 구조 파악 필요

#### C. QR 코드 생성
- **현황**: "qr" 탭에서 버튼 클릭 시 UUID 기반 QR 생성
  - QR 데이터는 고정 (무효화 불가)
- **UX 평가**:
  - 문제점: QR 생성 후 실제 인쇄 파일 다운로드 불가 → 외부 서비스에서 UUID→QR 변환 필요
  - 문제점: QR 유효성 관리 미흡 → 전단지 만료 후에도 QR은 유효한 상태

#### D. 통계 대시보드
- **현황**: "stats" 탭에서 다음 정보 표시
  - 전단지 수, 공유 수, 퀴즈 응시, 방문 수, 배분 포인트, 예산 잔액, 조회 수, 참여율
- **UX 평가**:
  - 장점: 포괄적 메트릭
  - **문제점 1**: "참여율"이 "퀴즈 응시 / 조회 수"로 정의 → 사실상 "퀴즈 응시율"만 봄
  - **문제점 2**: 포인트 "배분" vs "잔액" 비교가 없음 → ROI 계산 불가
  - **문제점 3**: 시간별/일자별 추이 그래프 없음 → 트렌드 파악 불가

#### E. 포인트 예산 충전
- **현황**: "charge" 항목에서 수동 충전
  - 1,000~10,000,000P 범위
  - 결제 수단 미구현 (수동 충전만 가능)
- **UX 평가**:
  - **문제점 1**: 실제 결제 통합 없음 → 영업팀이 수동 입금 처리 필요
  - **문제점 2**: 예산 부족 시 사용자 포인트 적립 차단 (에러 메시지만 표시) → 사업자가 긴급 충전하기 어려움

## 2. 사용자 메리트/디메리트 분석

### 메리트 (사용자가 이 앱을 쓰는 이유)

1. **낮은 진입장벽**: 게스트 모드로 로그인 없이 즉시 시작 가능
   - 당근마켓/토스와 유사한 "먼저 써보고 결정" 전략
   - 신뢰도 확보 전 강제 로그인 없음

2. **실질적 보상**: 포인트 → 기프티콘 교환 (카카오톡 발송)
   - 이론적 포인트가 아닌 "실제 상품"으로 교환
   - 카카오톡 기프티콘은 한국 사용자가 즉시 사용 가능

3. **게이미피케이션 요소**:
   - Canvas 기반 긁기 → 심리적 만족도 높음
   - 당첨 애니메이션(confetti) → 기대감 극대화

4. **오프라인 연동**: QR 방문 인증으로 실제 고객 유입 가능
   - "전단지 보는 것만으로 끝"이 아닌 매장 방문 유도

### 디메리트 (사용자가 이탈하는 이유)

#### 심각한 문제점

1. **포인트 적립 vs 교환 격차**
   - 전단지 1개 = 10P
   - 기프티콘 1개 = 3,000P (최소)
   - **결과**: 300개 긁어야 1개 교환 → "이게 맞나?" 의심심 발생
   - **비교**: 당근마켓의 리뷰 포인트는 즉시 사용 가능한 쿠폰으로 변환

2. **게스트 블로킹 메커니즘의 공격성**
   - 비로그인 1회 후 "2회차 차단" → 사용자가 "한 번 더 보고 싶다"는 욕구 강제 제거
   - **심리학**: Sunk Cost Fallacy 역이용 (이미 1회 사용했으니 로그인하세요) → 부작용으로 삭제
   - **개선안**: "3회 후 로그인 유도" 또는 "3회 제한" 같은 soft gate 필요

3. **기프티콘 한정성**
   - 12개 상품 고정 → "내가 원하는 브랜드가 없으면?" → 교환 불가
   - 비교: 토스 포인트는 50개+ 상품, 페이코 더드림은 직접 현금화
   - **결과**: "포인트가 쌓여도 쓸 곳 없음" → 이탈

4. **기프티콘 발송 시간**
   - "최대 1영업일" → 사용자 기대: 즉시 발송
   - 주말 신청 → 월요일~화요일 도착 (사실상 최대 3일)
   - **영향**: 기대와 실제의 갭 → 신뢰도 하락

5. **포인트 출금의 높은 진입장벽**
   - 최소 1,000P (5개 전단지)
   - 7일 경과 필수
   - 대기 중 중복 신청 불가
   - **결과**: "내 포인트가 언제 현금화되나?" 불안감

#### 중간 정도 문제점

6. **QR 1일 1회 제한의 불공평함**
   - 같은 가게 2회 방문 → 1회만 인정
   - 사용자 입장: "2번 가서 2번 돈 썼는데 왜 1번만?"
   - 비교: 신용카드 적립은 구매액 기반

7. **퀴즈 UI/UX 개선 부족**
   - 정답 공개 후 모달이 빠르게 닫힘
   - 사용자가 "뭘 맞혔는지 모르게" → "다음 문제 풀 동기" 감소

8. **다크모드 미지원**
   - MainPage에서 토글이 있지만 → ScratchCard 등 일부 컴포넌트에서 다크모드 고려 부족
   - 사용자: "밤에 쓰는데 눈이 아파" → 이탈

## 3. 자영업자 메리트/디메리트 분석

### 메리트 (사업자가 이 앱을 쓰는 이유)

1. **비용 효율성**
   - 종량제: 사용한 포인트(공유 수)만 차감
   - 고정비 없음 (기존 SNS 마케팅보다 저렴)
   - 비교: 구글/페이스북 광고는 확정 비용

2. **타겟 고객 확보**
   - 지역/카테고리 필터로 관심 고객만 노출
   - 비교: 전단지 뿌리기는 타겟팅 불가

3. **직접 고객 유입**
   - QR 코드 → 매장 방문 유도
   - 포인트 유인으로 실제 구매 유도 가능

4. **투명한 통계**
   - 전단지별 조회, 공유, 참여 수 추적 가능
   - 어떤 전단지가 잘 팔렸는지 데이터 기반 판단

### 디메리트 (사업자가 안 쓰는 이유)

#### 심각한 문제점

1. **온보딩의 복잡성**
   - "전단지 등록" 폼이 너무 길어서 (10+ 필드)
   - 사업자: "이걸 다 입력해야 하나?" → 포기
   - 비교: 당근마켓 가게 등록은 3단계, 5분

2. **포인트 예산 선결제 강제**
   - 사용자 1명이 공유 → 10P 차감 (즉시)
   - 사업자가 미리 1,000P 충전했는데 → "실제 도움이 된 고객이 몇 명일까?" 불명확
   - **비교**: Google Ads는 "클릭마다 비용" (확실한 action)
   - **심리**: "포인트가 사라졌는데 매출로 돌아왔나?" 의심

3. **의존도 높은 포인트 시스템**
   - 공유포인트 + QR포인트 2개 설정 → 어떤 걸 높게 해야 할까?
   - 포인트 너무 낮으면 → 사용자가 관심 없음
   - 포인트 너무 높으면 → 사업자 예산 빠름
   - **문제**: "포인트 책정" 자체가 사업자의 숙제

4. **결제 통합 미흡**
   - 포인트 충전이 "수동" (요청 → 영업팀 입금)
   - 자동 결제 시스템 없음
   - **심리**: "이 서비스 진짜 프로인가?" 신뢰도 하락

5. **고객 이탈 문제**
   - 게스트 블로킹 → 실제 고객 유입 감소
   - 기프티콘 한정성 → "이 가게 포인트로 뭘 살까?" 불명확
   - **결과**: "광고비 내고 고객은 못 얻음"

#### 중간 정도 문제점

6. **통계 분석의 부족함**
   - 참여율 = 퀴즈 응시율만 봄 → "공유는 몇 명?"을 못 봄
   - 시간별 추이 없음 → "어느 시간대 사용자가 많나?"를 모름
   - ROI 계산 불가

7. **QR 관리의 불편함**
   - QR 생성 후 "다운로드" 기능 없음 → 외부 서비스 필요
   - QR 무효화 불가 → 옛 전단지 QR도 유효

## 4. 구체적 개선 제안

### 🔴 Critical (즉시 수정 필요)

#### 4.1 포인트 경제 재설계
**현황**: 적립 vs 교환 격차가 너무 큼 (300:1)
**개선안**:
- sharePoint 최소값 올리기: 10P → 50P
- 기프티콘 최소가격 내리기: 3,000P → 1,500P
- **또는**: "포인트 부스터" 이벤트 (주말 2배 적립) 추가
- 파일: `src/pages/GiftShopPage.jsx` (GIFT_LIST 상품 포인트), `server/routes/flyers.js` (sharePoint 최소값)

#### 4.2 게스트 블로킹 UI 완화
**현황**: 2회차 "로그인 강제" 모달이 공격적
**개선안**:
- 비로그인 상태에서 "3회까지 자유" → 4회차부터 로그인 유도
- 로그인 유도 메시지: "로그인하면 이 포인트들이 살아나요!" (Soft nudge)
- 게스트 블로킹 모달 제거 (대신 DetailPage에서 subtle 배너)
- 파일: `src/App.jsx` (handleFlyerClick 로직), `src/pages/DetailPage.jsx`

#### 4.3 기프티콘 카테고리 확장 & 맞춤화
**현황**: 12개 고정 상품만 제공 → 사용자 선택지 부족
**개선안**:
- 상품 수 30개 이상으로 확대 (세부 카테고리 추가)
- "사용자 제휴사" 추가 → Lv2에서 100+ 브랜드 지원
- Admin에서 상품 CRUD 기능 추가
- 파일: `src/pages/GiftShopPage.jsx` (GIFT_LIST → DB 로드), `server/routes/gift.js` (신규 엔드포인트)

#### 4.4 QR 발송 시간 개선
**현황**: "최대 1영업일" → 실제로는 최대 3일
**개선안**:
- 시스템 통합: 카카오톡 Open API 사용 → "발송 즉시" (5분 이내)
- 또는: 발송 대기 시 "예상 시간" 표시 (투명성 확보)
- 파일: `server/routes/exchange.js` (기프티콘 발송 로직), 서드파티 API 통합

#### 4.5 사업자 온보딩 단순화
**현황**: 전단지 등록 폼이 너무 길어서 (10+ 필드)
**개선안**:
- "간단 등록" vs "상세 등록" 2가지 모드 분리
  - 간단: 매장명, 제목, 카테고리, 기간, 공유포인트만 (30초)
  - 상세: 이모지, 색상, 태그, 상품 목록 (3분)
- 폼 필드를 "단계"로 나누기 (Step 1/3)
- 파일: `src/pages/AdminPage.jsx` (폼 UI 재설계)

### 🟡 Important (1~2주 내 개선)

#### 4.6 포인트 출금 진입장벽 완화
**현황**: 최소 1,000P + 7일 경과 → 사용자 friction 높음
**개선안**:
- 최소 출금 금액 낮추기: 1,000P → 500P
- 소셜 로그인 + 휴대폰 인증 사용자: "3일 경과" 가능
- 7일 경과 전: "출금 예약" 기능 (리스트에 추가, 7일 후 자동 실행)
- 파일: `src/pages/MyPage.jsx` (출금 폼), `server/routes/withdrawal.js` (조건 변경)

#### 4.7 QR 방문 인증 규칙 개선
**현황**: 1일 1회 제한 → 사용자가 불공평하게 느낌
**개선안**:
- 규칙 변경: "1일 1회 → 1월 1회" (월별 누적 가능)
- 또는: 같은 가게 재방문 시 포인트 50% 적립 (완전 차감 아님)
- QR 바코드에 timestamp 포함 → "오전/오후 구분" 가능
- 파일: `server/routes/qr.js` (방문 검증 로직)

#### 4.8 퀴즈 UI 개선
**현황**: 정답 공개 후 모달이 빠르게 닫힘
**개선안**:
- QuizModal에 "정답 공개" 화면 3초 고정 (사용자가 볼 시간 확보)
- 정답 공개 시 "해설" 표시 (이 답이 왜 맞는지 설명)
- 오답 시 "다시 풀어보기" 버튼 추가
- 파일: `src/components/QuizModal.jsx`

#### 4.9 자영업자 통계 대시보드 고도화
**현황**: 기본 메트릭만 → ROI 계산 불가
**개선안**:
- "공유 수" 별도 표시 (현재는 "공유" + "방문" 합산)
- 일별 추이 그래프 추가 (지난 30일)
- ROI 계산: (배분 포인트 × 평균 포인트 가격) / (충전한 포인트) = %
- 파일: `src/pages/AdminPage.jsx` (stats 탭 UI 확장)

#### 4.10 자동 결제 시스템 추가
**현황**: 포인트 충전이 수동 → 사업자 불편
**개선안**:
- 토스페이/카카오페이 결제 모듈 추가
- "자동 충전" 기능: 포인트 1,000 이하 시 자동 충전
- 파일: `server/routes/business.js` (결제 엔드포인트), PG사 API 통합

### 🟢 Nice-to-have (1개월 이후)

#### 4.11 포인트 "실제 가격" 투명화
**현황**: 사용자가 포인트 가치를 모름
**개선안**:

- "리더보드" 추가: "이번 주 긁기 왕은?" (토스 신용점수 스타일)
- 파일: 신규 `src/pages/FriendsPage.jsx`, `server/routes/social.js` (업그레이드)
